⏺ Right — the payload transfer protocol. The mesh can send small packets (vouch attestations,
  presence, emergencies) but there's nothing for transferring the app itself. That's a file
  transfer layer on top of BLE.                                                                
   
  Here's what it needs:                                                                        
                                                                                             
  Protocol:
  A advertises: SERVICE_UUID + PAYLOAD_UUID characteristic
  B scans, sees "VexWallet available"
  B taps "I want this" → connects, reads PAYLOAD_UUID
    → gets metadata: {size, chunks, hash}
  B reads chunks sequentially: PAYLOAD_DATA_UUID[0..N]
  B reassembles, verifies SHA-256 hash
  B opens the HTML. Creates identity. Is now infrastructure.

  The math:
  - core.js (37KB) + secp256k1.js (14KB) + minimal UI ≈ 55KB raw
  - Minified ≈ 25-30KB
  - BLE MTU typically 512 bytes → ~60 packets
  - At BLE 4.2 speeds → ~2 min transfer
  - With BLE 5.0 (2Mbps) → ~30 sec

  What to add to core.js:

  meshPayloadAdvertise(htmlBlob)    — start offering the app
  meshPayloadDiscover()             — scan for nearby givers
  meshPayloadRequest(device)        — "I want this" (consent)
  meshPayloadReceive(onProgress)    — chunked download + verify
  meshPayloadAssemble(chunks, hash) — reassemble + integrity check

  The beautiful constraint: the payload IS the app that contains this code. So the transfer
  protocol transfers itself. The first person bootstraps from a URL or file share. After that,
  it's turtles all the way down.
